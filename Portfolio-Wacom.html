Overview

This project is about the wacom product drawing tablet 'driver', this enables the tablet to be used across various different applications (Photoshop and Illustrator as just some examples).

The current issue is the software for the driver is confusing, which is causing friction between the intended users and the software (which bleeds through to the drawing tablet as a result).

The aim of this project is to rectify this by Identifying pain points, adjusting according based on research and design principles and patterns while focusing on scalability to ensure this product can grow over time.

Architecture Understanding the Problem

The goal was to create an experience that aided artist and designers in their work rather than impede; imposed on their productivity while not intimidating novices or even professionals, common issues are:

  1. Clunky User Experience that confuses and frustrates market.
  2. Interactions of websites are uninspiring and offer nothing for an expensive piece of equipment.
  3. Information is thrown in without considering the user.

Artist tend to face these issues when using a pen display/tablet, therefore it is imperative that this issue is resolved to improve artist productivity.

Well the first part of any project is to do some Research, background Information on the topic to ensure one has a fundamental understanding of the issues out there (or existing solutions). This involved looking at the existing product that wacom has, identify the architecture framework and using Nielson Norman Group (NN/g), Material Design principles and conducting user research to outline the current issues before any ideation or solution focus design work is done.

Conducting questioners' to Identify problems then Interviews to figure out how to solve them was the first part of this project, to ensure we can figure out how our users thinks and behaves I construct a Persona and a Empathy Map to use a reference when It comes to building the User flow and Wireframe and everything else following that.

RocketDeveloping a Solution

At this stage when developing a solution to the problem uncovered, we need to do a couple things:

  1. Analyse the artifacts we created, if you are lucky to be on a team at this stage then you will find that through multiple lenses, you can understand the root of the problem very quickly, otherwise as an Individual it will take some time to reflect.

  2. Conduct some competitor research, looking at other company's allows you to see what others are doing right and wrong. Using online platforms you can hear others feedback based on comments which can help you get a gauge of how things are.

  3. Start to do some ideation work, rough sketches or wireframes, here the aim is to see what we currently understand and do not. Afterwards fill in any gaps in your knowledge, rinse and repeat.

Conclusion

Overalll this case study was a success due to the outcome that was achieved. Updating the the UX and UI to help artist in their task by not creating a system that acts as a block to their creative field, but deliveres on a software that helps enable their task.

Some of the interviews to create the persona could of been conducted better to avoid any leading questions, which was accidently done by priming users with an idea and leading them to the answers i perhaps wanted rather than what the user wanted (unintentially).

This however was addressed in follow up interviews and the data acquired through the initial interview was addressed by not anchoring back to it, and formatting my question in a neutral manner to avoid any issues with the data being bias.

However this project I was happy with the end result, even though i didn't necessarily follow material design patterns or NN/g in few areas because of the user feedback i received from testing in the early prototype stage.
